Project 3
Managers
--------
The SpriteManager stores textures in an unordered_map and sprites in a vector,
both valid for the SpriteManager's lifetime. GetSprite() returns a reference that
remains valid until vector reallocation occurs. SpriteIDs remain valid for the SpriteManager's lifetime, 
though storing sprite references is unsafe if RegisterTexture() 
may be called later.

The SoundManager stores sound buffers and sound instances in separate
unordered_maps, both valid for the SoundManager's lifetime. PlaySound() uses the
stored instances with non-blocking playback, allowing multiple sounds to play
simultaneously.

The WindowManager's GetWindow() returns a pointer valid for the WindowManager's
lifetime. The window is created in the constructor and destroyed in the
destructor, while camera settings persist until explicitly changed.

The EntityManager maintains EntityIDs that are valid until RemoveEntity() is
called, after which they may be reused. GetComponent() returns a reference valid
until the component or entity is removed. Component references are invalidated
by RemoveComponent() or RemoveEntity(), though component storage reallocation
is unlikely after initialization.

Systems
-------
The Input System processes keyboard input using SFML's scancode system for
tracking key presses and release states using bitsets.

The Movement System has velocity-based movement with acceleration and
deceleration, enforces maximum speed limits for horizontal movement, and
includes a fast-fall mechanic triggered by the S key while falling.

The Gravity System applies gravitational acceleration to entities with
GravityComponent and reduces gravity to 30% of normal when sliding on walls to
enable wall-sliding mechanics.

The Collision System uses AABB (Axis-Aligned Bounding Box) collision detection
with resolution that detects collision sides (top, bottom, left, right). It
handles both static and dynamic entities, implements star collection mechanics
where players collect stars for points, and manages star-to-star collisions
with momentum loss and star-to-wall collisions with bouncing and speed reduction.

The Camera System follows the player with zone-based movement. Hard zones at
the 10% edges cause the camera to immediately follow the player, soft zones at
the 30% edges smoothly tween to follow, and the center 40% zone slowly tweens
to follow. The camera also includes shake effects on player collision with walls
and handles world-to-window coordinate transformations.

The Sprite System renders sprites in either world space or screen space and
supports sprite positioning relative to entity locations.

The Spawn System spawns star entities at regular intervals with random
velocity directions and magnitudes, using configurable spawn rates and total
spawn counts.

The Score System tracks score with a digit-based display, updates sprite
components for score visualization, and awards 10 points per star collected.

DESIGN DECISIONS
---------------
Freed entity slots are reused to minimize memory allocation. Components are
stored in vectors with entity-to-index mapping for efficient access. Some systems
use static variables for jump state tracking so its easier to keep track of over many frames. 
Delta time is currently fixed at 1/60 seconds, assuming a stable 60 FPS frame rate. 
This provides a consistent time step for updates, 
ensuring that gravity and acceleration calculations do not rely on frame rate so we can avoid 
the game being buggy at higher or lower framerates.
For collision detection, we decided to use tileW and tileH from SpawnComponent to create AABB bounds 
for an easy work around.
We used Doxygen for the api documentation.
We also made it so that stars cannot collide with spawners for spawning simplicty.
And lastly, we reuse some code from our Project 1 Pt.2, this ranges from inspiration for collision 
structure, to almost our exact MathUtil.h. This means that some helper functions in MathUtil.h
are not used for this project but will be left there for contingency purposes in the future.

Project 4
Managers
--------
The SpriteManager stores textures in an unordered_map and sprites in a vector,
both valid for the SpriteManager's lifetime. The atlas is finalized after map
load to avoid reallocations; SpriteIDs remain valid for the lifetime.

The SoundManager stores sound buffers and sound instances in separate
unordered_maps, both valid for the SoundManager's lifetime. PlaySound() uses the
stored instances with non-blocking playback, allowing multiple sounds to play
simultaneously.

The WindowManager's GetWindow() returns a pointer valid for the WindowManager's
lifetime. The window is created by ECSEngine; camera settings persist until
explicitly changed.

The EntityManager maintains EntityIDs that are valid until RemoveEntity() is
called, after which they may be reused. GetComponent() returns a reference valid
until the component or entity is removed. Component references are invalidated
by RemoveComponent() or RemoveEntity().

The ShaderManager caches shaders (hurt flash, rock shield). The SystemManager
holds and runs systems per scene. ECSEngine owns one window and a scene stack;
only the top scene runs each frame and the engine does a single display().

Scenes
------
MainMenuScene: draws “Leo the Wizard Cat” and “Press Enter to Start”; completes
on Enter to reveal gameplay.

GameplayScene: full cat/slime platformer; Escape edge-press pushes PauseScene.

PauseScene: dark overlay with “Paused” and resume prompt; Enter/Esc pops it.

Systems
-------

InputSystem applies buffered input to movement/jump/dash/wall-slide intents.

SpellSystem handles element select (1–4) and casting (E), spawns projectiles based on where we are looking ie the last directional input. This is
also the system that manages the sounds when we swap to a different element and when we fire said element. It tracks which element we were on.

ProjectileSystem manages the actual spell flying and colliding. So when we send a spell out they become a projectile and when they collide it depends ont this
system to dictate the explosion, ie the sounds and animations for the specific spell

EnemySystem manages slimes (movement, damage flash, death, resistances). This means that this is what has the component and manages what happens 
to slimes when hit, what resistances they have, and it does use the hpSystem in tandem to track the slime hp. The movement here is just random left or right
movement where we will randomly swap directions while moving.

HpSystem tracks health, invulnerability timers, dying state, and rock shield. 

CheckpointSystem/Campfire manages checkpoints and saves.

CameraSystem was changed to a lookahead system instead so depending on where the player is going thats where more of the screen realestate will be.

AnimationSystem plays and stores animations for player, enemies, spells, explosions.

SpriteSystem renders atlas sprites (world/UI), supports flip and shaders for
damage/shield, and parallax per layer. we have layers -4 to 4, where 1 and 2 are the neutral 1.0 layers with no parallax.

SpawnSystem handles map-driven enemy/entity spawns.

ProcessEventsSystem collects all SFML window events and updates global input buffers so other systems can react frame-accurately.

CollisionUpdateSystem updates translated AABBs for all collidable entities and prepares broad-phase collision lists used by CollisionSystem.

ShaderManager loads and caches GLSL fragment shaders (hurt flash, rock shield) and exposes them so SpriteSystem can apply shaders per entity.

SpellSystem listens for casting input, handles element switching, spawns ProjectileComponent entities with inherited spell properties

ProjectileSystem moves projectiles, checks collision, triggers explosion animations, and removes expired projectiles.

EnemySystem controls AI movement, updates state, interacts with HpSystem for damage/death, plays sounds on hit/death.

HpSystem applies damage, handles invulnerability, regenerates shield, triggers death/despawn events.

CheckpointSystem activates campfires when touched, updates respawn location, and resets the scene after death.

ScoreSystem was supposed to add scores when enemies are defeated, but we didn't have time to do so.

DESIGN DECISIONS
---------------
Freed entity slots are reused to minimize allocation; components use vector
storage with entity-to-index mapping. Fixed timestep assumption ~1/60s; Scene
clamps dt to 1/30s max to prevent physics spikes. Scene stack isolates
menu/pause/gameplay with one display() in ECSEngine. Parallax keeps gameplay
layers at 1.0 to align sprites and colliders; backgrounds use <1 for depth.
Projectiles have a 0.1s grace period to avoid self-collision and enforce
lifetime-based removal. Atlas is finalized once after map load to lock sprite
rects/IDs. We reuse prior helpers (EntityManager, MathUtil) and extend with
spells, enemies, hp/checkpoint, and scenes. The goal post is indicated with a sign to go next,
a portal that goes to the next level is pending for next implementation.
